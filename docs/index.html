1. How we evaluated — a real-news benchmark

113 real statements from Yahoo Finance across stocks, macro, commodities, FX, indices, rates, crypto and 5 claim types. Factual labels are cross-checked against SEC/FRED; a perturbed copy (changed number/date) gives a matched false. Opinion/forecast carry no truth value (they test classification).

Why real news, not figures from a database: the job is to judge whether a real-world statement is accurate. Perturbing numbers taken from the same database the agent queries would be too easy and overstate performance — so we evaluate on real statements, with labels verified against the primary sources.

The agent almost never confirms a true claim (1/50) and can't separate true from false (AUC ≈ 0.34) — decent at flagging false claims, but effectively useless at confirming a true one.

3. Where the errors come from — multiple causes, no single root cause

We attributed all 50 true-claim failures on real news to a mechanism:

  1. a. It couldn't get the data — 44% (the biggest cause). Which data: macro (CPI, payrolls), commodities (Brent, copper), indices (S&P 500, Dow), FX (EUR/USD), plus some misrouted stock. Why — and it is NOT a missing key (SEC needs none; FRED is configured):
    • Source not implemented: the system declares 25 data sources (BLS, BEA, ECB, CFTC…) but only ~12 have real fetchers — the rest are menu "cards" with no code, so non-equity has almost no working source except FRED.
    • Routing too narrow: FRED actually has CPI/payrolls/WTI/S&P 500/EUR-USD, but its keyword map only knows "cpi"/"inflation"/"gdp" — it never matches "consumer prices", "payrolls", or "Brent", so it never queries.
    So the data usually exists in a covered source but is not wired/routed — it is rarely that the data genuinely doesn't exist (only crypto truly lacks a source).
  2. b. It matched the wrong time period — 10% (a bug we found and fixed, "Fix A"). SEC reports overlapping periods (quarter/half-year/annual); the check compared a true claim to the wrong one → contradicted it. Fix A (surface values tagged by period) resolved this class — but it is only ~10% of real failures, so real-news accuracy stayed ≈35%: the bottleneck is elsewhere (data coverage).
  3. c. It found the data but wouldn't commit to "true" — systemic. Even when the figure matches, the verdict caps at partially_supported and almost never reaches supported (true confirmed 1/50). The verdict logic is too conservative.
  4. d. Misclassification & concept gaps. Factual statements labeled forecast/opinion → never fact-checked (9 claims); segment metrics (iPhone revenue, AWS) aren't mapped SEC concepts → not retrievable (11 claims); some entities (Nasdaq, GBP/JPY) not extracted (3 claims).

4. Does AI help? (per stage)

  • Entity extraction — yes (80.5% → 89.4% with the LLM).
  • Classification — n/a (rule-based; AI not used).
  • Verdict — AI was not the bottleneck; the failures are data coverage and routing, not the model. AI is necessary but not sufficient here.

5. Limitations

  • Verdict accuracy is trustworthy mainly for US equities (SEC); non-equity coverage is limited by unimplemented sources + narrow routing, not by keys.
  • The system advertises 25 data sources but only ~12 are implemented — much of the "coverage" is a menu without a backend.
  • Opinion/forecast have no ground truth (we only check classification + routing); crypto/exotic facts are out of scope.

6. Bottom line

The front end (entity extraction, classification, routing) works reasonably and AI improves extraction. The verdict stage is unreliable on realistic claims, but there is no single root cause: ~44% of failures are "couldn't get the data" (sources not implemented or not routed — not a key problem), ~22% are concept gaps, only ~10% are the period bug we fixed, and a systemic conservatism keeps it from confirming true claims. The reliability bottleneck is data coverage, not one bug.
