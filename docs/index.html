1. How we evaluated — a real-news benchmark

113 real statements from Yahoo Finance across stocks, macro, commodities, FX, indices, rates, crypto and 5 claim types. Factual labels are cross-checked against SEC/FRED; a perturbed copy (changed number/date) gives a matched false. Opinion/forecast carry no truth value (they test classification).

Why real news, not figures from a database: the job is to judge whether a real-world statement is accurate. Perturbing numbers from the same database the agent queries would be too easy and overstate performance — so we use real statements with verified labels.

The agent almost never confirms a true claim (1/50) and can't separate true from false (AUC ≈ 0.34) — decent at flagging false claims, but effectively useless at confirming a true one.

3. Where the errors come from — multiple causes, no single root cause

We attributed all 50 true-claim failures to a mechanism:

  1. a. It couldn't get the data — 44% — it never queried a source it already has. Not a missing key, not a missing source: (1) routing too narrow (≈13: macro/commodity/index/FX) — never sent to FRED, though FRED has CPI/payrolls/WTI/S&P 500/EUR-USD; (2) misclassified → retrieval skipped (≈9: stock) — labeled forecast/opinion, so SEC was never queried.
  2. b. Wrong time period — 10% (the bug we fixed, "Fix A"). SEC has overlapping periods (quarter/half-year/annual); it compared a true claim to the wrong one → contradicted it.
  3. c. Found the data but wouldn't commit to "true" — systemic. Even when the figure matches, the verdict caps at partially_supported, almost never supported.
  4. d. Misclassification & concept gaps. Facts labeled forecast/opinion (9); segment metrics like iPhone/AWS revenue not mapped (11); some entities not extracted (3).

4. Improvements we made (routing + Fix A) — and the result

Fix A — period-aware SEC facts. The fact extractor kept only the 2 most-recently-filed values per metric, so a claim's specific fiscal-period value was often absent → not_found → escalated to contradicted. Fix: surface full-year and quarterly figures tagged by reporting period, so the claim's period value is present.

Routing fix — send macro/commodity/index/FX to FRED. Both routing layers (source selection + the FRED keyword map) only knew a few terms. Fix: expand both with EN/ZH synonyms/aliases ("consumer prices", "Brent", "S&P 500", "EUR/USD", …) so these claims route to FRED, which has the series.

Neither fix moved real-world accuracy. Each repaired a real layer (Fix A removed period-mismatch contradictions; the routing fix now retrieves the previously-missing data), but the verdict stayed ≈36% because the next layer immediately blocks it:
  • FRED fetch is date/period-blind (the root of "retrieved but matched the wrong value"): fred() always returns the latest raw observation, never the claim's date or a derived value. So "S&P 500 closed at 7,501 on May 15" is checked against FRED's latest (7,473 on May 22), and "CPI rose 3.8%" against the index level (332), not the year-over-year %.
  • CPI / payrolls are still misclassified → skipped before retrieval.
The failures are a stack — routing → date/period-blind fetch → classification → conservative verdict. Fixing one layer just exposes the next: the strongest evidence that there is no single root cause.

5. Does AI help? (per stage)

  • Entity extraction — yes (80.5% → 89.4%).
  • Classification — n/a (rule-based).
  • Verdict — AI was not the bottleneck; the failures are routing, fetch-alignment and classification, not the model.

6. Bottom line

The front end works reasonably and AI improves extraction. The verdict stage is unreliable on realistic claims, with no single root cause: ~44% "never queried a source it already has", ~22% concept gaps, ~10% the period bug, plus systemic conservatism. We fixed two real layers and real-world accuracy did not move (≈36%) — each fix exposes the next problem. The bottleneck is the depth of the data/verdict pipeline, not one bug.

7. Limitations & future work

  • On this benchmark, verdict failures are routing / fetch-alignment / classification problems (the data is reachable), not missing data or keys. Opinion/forecast have no ground truth; crypto/exotic out of scope.
  • Future work (priority order): (1) make fred() date/period-aware (observation_start/end; units=pc1 for "rose X%") — the root fix for "retrieved but wrong value"; (2) fix the factual-vs-forecast classifier; (3) implement the declared-but-missing sources (BLS/BEA/ECB/…), lower priority.
